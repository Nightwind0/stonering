Weapons, Armor, Items, Rings, Runes, and Key Items should all be loaded from resource files.

Weapons will include the image associated with them, their type, color class, modifiers (such as rune, black, etc), and stats such as power, price, etc. The game engine will take care of all the rules, including damage upon enemies and casting the appropriate spells, etc.

Armor is similarly done. As are Rings.

 

Items contain the image associated with it, the name, and a collection of attribute modifiers that decribe what the item does. They would also have a price, and an indicator as to whether they can be used in battle, or in the menu, or both.  They would also have an english description of the use.

 

Runes have a name, an image, and the spell # the rune ties to. They would also have a high price.

 

Key Items include a name and an optional image. They also have an optional  script which runs when the item is used from the menu.

 

 

Ams: Attribute Modifiers: these are simple structures which describe changes to HP, MP, and status effects. They can be used to describe an attack, a cure, and a change of status. Items use these to describe what effect they have. Thus we could create a cure potion just as well as a bomb to be used on the enemy.

 

 

“Dead” should be a long-term status effect.

 

 

 

Displaying:

All objects which can be displayed should display themselves.

The Battle should be an object which knows how to display itself and can do so at any screen co-ordinate, so that battle screens can scroll on and off screen.

The same is true for the WorldMap.

•When going from one place to another via a transport script, (such as from the worldmap into a town or vice versa) the new map should slide in from one of the 4 (or even 8) directions and cover up the old. This would be a neat transition.

Another neat transition might be to have the next view either unroll like a scroll or the previous scene roll up like a map. This would probably be harder but the effect would rock!

* The text which comes from scripts in the WorldMap (as well as in battle) should be handled by a TextHandler object and know how to draw itself. It should slide down from the top, and display 3 to 5 lines at a line, and when the button is pressed, it should scroll one? Line at a time.  It should also know how to draw any menu that the script should request. The engine should pass key presses to the text bar when it is being shown, so that  scrolling, menu choosing, and ending of the text bar can occur

 

 

Also, there should be “Display” objects, which represent things that are displayed for a period of time and then dissapear, such as the points resulting from an attack in a Battle (the white/ green HP numbers besides the target of the attack/spell/item usage). These are created, given an X,Y to start at (I say that because certain displays might move around), and left to display until such time that they destroy themselves (or somehow request to be destroyed). That way, the numbers will float in the air until they fade away all by themselves, etc. The same goes for the animation of a spell, or an attack or what have you. It's just an object that floats over the target and displays itself, possibly many different  frames. OR the game may go into a separate function to display these animations and be basically “modal” and then return to the regular loop.

Displaying Levels:

                     We give the LevelMap object an x and a y as to where in the level to display. Note that we DO NOT choose co-ordinates by tiles. Tiles are used only in building the level to be displayed. Remember, we want smooth scrolling.. and smooth movement (walking). So the LevelMap object will calculate which tiles to show, and request them to be drawn at a certain point. ALSO, LevelMap's draw should also be able to accept an optional point of origin (top-left) so that levels can slide in from different directions as a cool effect.

 

NPC Movement:

Basically just implement a few basic NPC  walk algorithms and choose which one to use when designing an NPC in the level design program.  (Basically you write into the load script a command to change movement algorithm)

*** Scripts should be able to change the movement algorithm, such that an NPC might get freaked out and change to Crazed, or might already be panicked and when you return with good news they change to Calm. Or a stuck person (None) could become unstuck (Calm) or what have you.

•Only on-screen objects should be animated.

•Ideas for NPC alogorithm choices

• 80's Robot – Walks one direction until hits an obstacle, then turns 90 degrees and walks that way... rinse & repeat. Useful for someone just on a casual stroll. Good alternative to “Calm” for people who you don't mind if they wander a lot.

• Speedy – Runs one direction – hits a wall, turns around and runs back. Hyper dog?? Stupid kid?? Panicked person?

• Sporadic – Only goes in one direction for maybe 4 tiles max, then picks another way and goes. At an obstacle, picks a random direction of the remaining 3. Could be used for animals....

• None – doesn't move. Second most common probably.

• Calm – Most should use this. Occasionally takes a couple steps in one direction or another. At most 3 tiles. Then waits around for a while before walking again in some random direction.

•Crazed – Sporadic, only running instead of walking.



There should only be two speeds – walk and run.

Anything that can be a still image (tiles, monsters, etc) should also be able to be an animation. They should be interchangeable, and there needs to be * A WAY TO DEFINE SIMPLE ANIMATIONS (FOR TILES, MONSTERS, ETC)  IN THE RESOURCE FILES

 

*** A good way to implement animations, would be to have quantums of time, and for each quantum of time you select an image to be displayed.

                                          Time 1 - - Image 1

                                          Time 2 –  Image 1

                                          Time 3 – Image 1

                                          Time 4 – Image 2

                     Thus, it stays usually image 1, but blinks to image 2. This is similar to musical beats. You can also leave a time slot blank to show nothing at all, thus you could have something flashing....  And if you just want one solid image the whole time (as is usually the case) then you simply put

                                          Time 1 – Image 1

                     and it will just cycle that one over and over, appearing as a solid image on screen.

 

** We need to be able to have animations with cycles at least 4 seconds long. For example, waves on the beach which come in and out in 2 second intervals.

                                          Time 1 – Wave Out

                                          Time 2 -  Inbetween (Closer to Out)

                                          Time 3 – Inbetween (Closer to in)

                                          Time 4 – Wave In

 

 

---- Level Editor

You move either a selection rectangle, or a simple one-tile cursor around what I will refer to as the “map”

you can move around the map using the arrow keys.

 

 There should be a placement mode, in which you move around the map and hit space on the tiles where you want to place your selected tile AND

There shoul d be a selection mode, where you click and drag a selection rectangle over the tiles you want to select, and you can then set them to all one tile, or delete them, or whatever.

Hitting insert should toggle between placement and selection.

* You change a tile just by placing a different tile over it. It simply replaces it.

Hitting ENTER rather than space over a square opens up the object edit/create screen, showing a list of all objects (if any) that currently have the X and Y values matching where you hit enter. If none in the list, it opens up in create mode, and sets the default X and Y values to the tile where you hit enter.

Pressing F1 shows you a list of all the objects in the level, and you can select one to display/edit by hitting enter on it. That opens up the edit screen for that particular object. Escape leaves the screen. 'S' saves the object.

 

*Editing Objects:

There will be a selection of pre-made objects, where you only need fill out certain fields.

For example,  a treasure box pre-made would use the treasure-box icon, and automatically generate an action-button script which calls GiveItem() provided that the first memory word doesn't indicate that it's already been opened. Furtheremore, it generates a display script which chooses either the opened or closed chest icon for display. You only need tell it which item it should give.

Also, door  pre-mades can be done, as well as transportation tiles, standard NPCs, etc.

Of course, custom objects can be made

Afterwards you press S to save the object and it leaves the object-edit screen.

Tile Selection:  at the bottom of the screen is a row of tiles, one of which is highlighted with a cursor of some sort.

You move the cursor left and right over the row, and whichever you leave it on is the selected tile. This basically is the method for selecting a column in a matrix of tiles. The divide  and multiply keys on the num-pad  will be responsible for this.

You can access another row from the matrix by hitting minus (up) and plus (down) on the numpad. This basically is the method for selecting a row in a matrix of tiles.

You change matrices by hitting Page Up/Page Down.
